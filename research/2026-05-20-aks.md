## Executive Summary

Azure Kubernetes Service (AKS) is Azure's managed Kubernetes offering, providing automated control-plane management while users pay only for agent nodes (VMs). For the Zava Smart Order Feasibility demo — a small LangGraph Python agent exposed publicly via the A2A protocol — AKS provides a simple, well-documented hosting path with a **Free tier** control plane and a 1–2 node `Standard_D2s_v5` node pool.

The recommended architecture for this demo is: **Free-tier AKS cluster** with a single system node pool of 1–2 nodes, the **Application Routing add-on** (managed NGINX ingress) for public endpoint exposure with **trusted HTTPS/TLS** via a CA-issued certificate in Azure Key Vault + Azure DNS hostname, **Workload Identity** for pod-to-Azure authentication (to Foundry, Key Vault), and **Azure Container Registry (ACR)** for image storage using `az acr build` for simplicity. Container Insights with a Log Analytics workspace provides observability; Managed Prometheus is optional and not included in the default deployment.

**Estimated cost:** approximately **$5–8/day** in East US depending on node count (1 node ≈ $4.21/day; 2 nodes ≈ $6.51/day) — inclusive of Free-tier control plane, node VMs, Standard Load Balancer, public IP, ACR Basic, Log Analytics ingestion, and Azure DNS zone. Node VM pricing is approximate — always verify with the [Azure Pricing Calculator](https://azure.microsoft.com/en-us/pricing/calculator/) for your offer type. Always verify the Kubernetes version at deployment time with `az aks get-versions` since AKS versions rotate on a 12-month cycle.

This report covers the eight research questions: AKS control-plane SKUs and Kubernetes versions, Bicep deployment snippets, ingress options, workload identity, image build/push, observability, security defaults, and cost estimates — all sourced from official Microsoft Learn documentation and Azure pricing pages.

## 1. Overview

### What It Is

Azure Kubernetes Service (AKS) is a managed Kubernetes container orchestration service on Azure. Microsoft manages the Kubernetes control plane (API server, etcd, scheduler, controller manager); users provision and pay for the worker (agent) node VMs.

> "Azure Kubernetes Service (AKS) simplifies deploying a managed Kubernetes cluster in Azure by offloading the operational overhead to Azure."
> — Source: [What is Azure Kubernetes Service (AKS)?](https://learn.microsoft.com/en-us/azure/aks/what-is-aks)

### Why It Matters (for this demo)

The Zava demo requires hosting a **LangGraph Python agent** that is publicly accessible (with auth) so the Foundry V2 agent can reach it via A2A. AKS provides:
- Production-grade Kubernetes with managed control plane
- Built-in NGINX ingress (Application Routing add-on)
- Workload Identity integration for pod-to-Azure-resource authentication
- ACR integration for container image management
- Container Insights for logs and monitoring

### Key Features (demo-relevant)

- **Free tier** control plane — no cluster management fee for demos/dev
- **Application Routing add-on** — managed NGINX ingress controller, no extra infra
- **Workload Identity** — pods authenticate to Azure services (Foundry, Key Vault) via Entra ID federated credentials
- **ACR attach** — cluster pulls images from ACR without managing Docker credentials
- **Container Insights** — logs shipped to Log Analytics automatically
- **Bicep-native deployment** — full IaC support via `Microsoft.ContainerService/managedClusters`

## 2. Key Concepts — AKS Basics in 2026

### 2.1 Control-Plane Pricing Tiers

AKS offers three pricing tiers for cluster management ([AKS Free, Standard, and Premium Pricing Tiers](https://learn.microsoft.com/en-us/azure/aks/free-standard-pricing-tiers)):

| Tier | Cluster Management Cost | SLA | Max Nodes | Use Case |
|------|------------------------|-----|-----------|----------|
| **Free** | $0 | No SLA (best-effort, ~99.5% internal goal) | 1,000* | Dev/test, demos |
| **Standard** | ~$0.10/cluster/hour (~$72/month) | 99.9–99.95% uptime SLA | 5,000 | Production |
| **Premium** | ~$0.60/cluster/hour (~$432/month) | 99.9–99.95% SLA + LTS | 5,000 | Mission-critical + LTS |

*\*Not recommended above 10 nodes at Free tier.*

> "AKS offers three pricing tiers for cluster management: the **Free tier**, the **Standard tier**, and the **Premium tier**."
> — Source: [Free, Standard, and Premium pricing tiers for AKS](https://learn.microsoft.com/en-us/azure/aks/free-standard-pricing-tiers)

> "Node limit is 1,000 (but not recommended at more than 10 node scale) when operating without an SLA in Free tier."
> — Source: [AKS pricing](https://azure.microsoft.com/en-us/pricing/details/kubernetes-service/)

**Recommendation for Zava demo:** Use the **Free tier**. No SLA is needed for a demo, and it eliminates the ~$72/month control-plane fee.

### 3.2 Minimum AKS + ACR Bicep Snippet

The following Bicep deploys a small AKS cluster with:
- Free tier, system-assigned managed identity
- Workload identity + OIDC issuer enabled
- Application Routing add-on (managed NGINX)
- ACR with AcrPull role assignment for the cluster
- Container Insights via Log Analytics
- Azure DNS zone for the demo hostname
- Key Vault for TLS certificate storage

```bicep
// Source: Adapted from official quickstart and Bicep reference
// https://learn.microsoft.com/en-us/azure/aks/learn/quick-kubernetes-deploy-bicep
// https://learn.microsoft.com/en-us/azure/templates/microsoft.containerservice/2026-02-01/managedclusters
// Provenance: synthesized — combines patterns from multiple official sources

@description('The name of the AKS cluster')
param clusterName string = 'aks-zava-demo'

@description('Location for all resources')
param location string = resourceGroup().location

@description('DNS prefix for the AKS cluster')
param dnsPrefix string = 'zava'

@description('Number of agent nodes')
@minValue(1)
@maxValue(3)
param agentCount int = 2

@description('VM size for agent nodes')
param agentVMSize string = 'Standard_D2s_v5'

@description('Name of the Azure Container Registry')
param acrName string = 'acrzavademo'

@description('Name of the Log Analytics workspace')
param logAnalyticsName string = 'law-zava-demo'

@description('Name of the Azure Key Vault for TLS certificate')
param keyVaultName string = 'kv-zava-demo'

@description('Custom DNS zone name for the demo (e.g., zava-demo.example.com)')
param dnsZoneName string

// ---------- Log Analytics Workspace ----------
resource logAnalytics 'Microsoft.OperationalInsights/workspaces@2023-09-01' = {
  name: logAnalyticsName
  location: location
  properties: {
    sku: {
      name: 'PerGB2018'
    }
    retentionInDays: 30
  }
}

// ---------- Azure Container Registry ----------
resource acr 'Microsoft.ContainerRegistry/registries@2023-07-01' = {
  name: acrName
  location: location
  sku: {
    name: 'Basic'
  }
  properties: {
    adminUserEnabled: false
  }
}

// ---------- Azure Key Vault (for TLS certificate) ----------
resource keyVault 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: keyVaultName
  location: location
  properties: {
    sku: {
      family: 'A'
      name: 'standard'
    }
    tenantId: subscription().tenantId
    enableRbacAuthorization: true
  }
}

// ---------- Azure DNS Zone ----------
resource dnsZone 'Microsoft.Network/dnsZones@2023-07-01-preview' = {
  name: dnsZoneName
  location: 'global'
}

// ---------- AKS Cluster ----------
resource aks 'Microsoft.ContainerService/managedClusters@2026-02-01' = {
  name: clusterName
  location: location
  sku: {
    name: 'Base'
    tier: 'Free'
  }
  identity: {
    type: 'SystemAssigned'
  }
  properties: {
    dnsPrefix: dnsPrefix

    // Enable OIDC issuer for workload identity
    oidcIssuerProfile: {
      enabled: true
    }

    // Enable workload identity
    securityProfile: {
      workloadIdentity: {
        enabled: true
      }
    }

    // Enable Application Routing add-on (managed NGINX)
    ingressProfile: {
      webAppRouting: {
        enabled: true
      }
    }

    // System node pool
    agentPoolProfiles: [
      {
        name: 'system'
        count: agentCount
        vmSize: agentVMSize
        osType: 'Linux'
        mode: 'System'
        osDiskSizeGB: 30
      }
    ]

    // Container Insights
    addonProfiles: {
      omsagent: {
        enabled: true
        config: {
          logAnalyticsWorkspaceResourceID: logAnalytics.id
        }
      }
    }
  }
}

// ---------- ACR Pull Role Assignment ----------
// Grants the AKS kubelet identity AcrPull on the ACR
var acrPullRoleId = '7f951dda-4ed3-4680-a7ca-43fe172d538d' // AcrPull built-in role

resource acrPullAssignment 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
  name: guid(acr.id, aks.id, acrPullRoleId)
  scope: acr
  properties: {
    principalId: aks.properties.identityProfile.kubeletidentity.objectId
    roleDefinitionId: subscriptionResourceId('Microsoft.Authorization/roleDefinitions', acrPullRoleId)
    principalType: 'ServicePrincipal'
  }
}

// ---------- Outputs ----------
output clusterName string = aks.name
output clusterFqdn string = aks.properties.fqdn
output oidcIssuerUrl string = aks.properties.oidcIssuerProfile.issuerUrl
output acrLoginServer string = acr.properties.loginServer
output keyVaultUri string = keyVault.properties.vaultUri
output dnsZoneId string = dnsZone.id
```
> — Source: [Quickstart: Deploy an AKS cluster using Bicep](https://learn.microsoft.com/en-us/azure/aks/learn/quick-kubernetes-deploy-bicep) + [Microsoft.ContainerService/managedClusters 2026-02-01](https://learn.microsoft.com/en-us/azure/templates/microsoft.containerservice/2026-02-01/managedclusters) | Provenance: synthesized

## 4. Core Usage — Ingress & Public Endpoint

### 4.1 Ingress Options Comparison

AKS provides several ways to expose services publicly ([Ingress Networking in AKS](https://learn.microsoft.com/en-us/azure/aks/concepts-network-ingress)):

| Option | Type | Managed? | Cost | Complexity |
|--------|------|----------|------|------------|
| **Application Routing add-on (NGINX)** | L7 ingress | Yes (AKS managed) | No add-on charge (LB + public IP billed separately) | Low |
| **Application Gateway for Containers** | L7 gateway | Azure-hosted | Separate billing | Medium |
| **Plain LoadBalancer Service** | L4 | Yes (Azure LB) | Standard LB cost | Lowest |
| **Azure Front Door + AKS** | Global L7 | Azure-hosted | Separate billing | High |

> "The application routing addon is the recommended way to configure an Ingress controller in AKS. The application routing addon is a fully managed ingress controller for Azure Kubernetes Service (AKS) that provides the following features: Easy configuration of managed NGINX Ingress controllers based on Kubernetes NGINX Ingress controller."
> — Source: [Ingress Networking in AKS](https://learn.microsoft.com/en-us/azure/aks/concepts-network-ingress)

### 4.2 Recommendation: Application Routing Add-on

For the Zava demo, the **Application Routing add-on** is the best choice because:
- It's managed by AKS — no need to install/maintain NGINX yourself
- It supports TLS termination via Azure Key Vault integration
- It supports custom domain names via Azure DNS integration
- It's fully GA and supported through November 2026

**Important cost note:** The Application Routing add-on itself has **no add-on charge** — the NGINX ingress controller runs as pods within your cluster. However, the add-on creates a **Standard Load Balancer** and a **public IP address** for external traffic, which are billed at standard Azure networking rates ([Azure Load Balancer pricing](https://azure.microsoft.com/en-us/pricing/details/load-balancer/) and [Public IP pricing](https://azure.microsoft.com/en-us/pricing/details/ip-addresses/)). These costs are included in the cost estimate in Section 9.

> "The application routing add-on with NGINX delivers the following: Easy configuration of managed NGINX Ingress controllers based on Kubernetes NGINX Ingress controller."
> — Source: [Managed NGINX ingress with the application routing add-on](https://learn.microsoft.com/en-us/azure/aks/app-routing)

**Important note on NGINX retirement:** The upstream Ingress NGINX project is retiring with maintenance ending March 2026. However, Microsoft provides official support for the application routing add-on NGINX through **November 2026** ([Ingress Networking in AKS](https://learn.microsoft.com/en-us/azure/aks/concepts-network-ingress)). For a demo in mid-2026, this is still fine. For long-term production, Microsoft recommends migrating to the Gateway API implementation.

> "Production workloads remain fully supported through November 2026. Migrate to the application routing Gateway API implementation for a Gateway API-based ingress traffic management experience."
> — Source: [Ingress Networking in AKS](https://learn.microsoft.com/en-us/azure/aks/concepts-network-ingress)

### 4.4 Trusted TLS for the Public A2A Endpoint

**All public endpoints MUST use HTTPS with a trusted CA-issued certificate.** For an external customer demo with public endpoints, self-signed certificates are not acceptable — they require disabling certificate verification on the client, which undermines TLS security guarantees.

> "Self-signed certificates are digital certificates that aren't signed by a trusted third-party CA. The company or developer responsible for the website or software creates, issues, and signs these certificates. This is why self-signed certificates are considered unsafe for public-facing websites and applications."
> — Source: [Set up a Custom Domain Name and SSL Certificate with the Application Routing Add-on](https://learn.microsoft.com/en-us/azure/aks/app-routing-dns-ssl)

The recommended path for this demo is **Application Routing + Azure Key Vault + Azure DNS + CA-issued certificate**. Azure Key Vault has trusted partnerships with Certificate Authorities including DigiCert and GlobalSign for automated certificate provisioning:

> "Azure Key Vault has a trusted partnership with the following Certificate Authorities: DigiCert, GlobalSign."
> — Source: [Integrating Key Vault with DigiCert certificate authority](https://learn.microsoft.com/en-us/azure/key-vault/certificates/how-to-integrate-certificate-authority)

**Certificate options (choose one):**
- **DigiCert or GlobalSign** — integrate directly with Azure Key Vault for automatic issuance/renewal
- **Any other CA** — obtain a certificate (PFX) and import it into Key Vault manually
- **Let's Encrypt** — free, automated via ACME; requires a cert-manager ClusterIssuer in the cluster (not built into Application Routing, but well-documented for AKS)

**Step 1: Set up Azure DNS zone and integrate with Application Routing.**

```bash
# Create an Azure DNS zone (if you don't already have one)
az network dns zone create \
  --resource-group rg-zava-demo \
  --name zava-demo.example.com

# Get the DNS zone resource ID
ZONE_ID=$(az network dns zone show \
  --resource-group rg-zava-demo \
  --name zava-demo.example.com \
  --query "id" --output tsv)

# Integrate the DNS zone with Application Routing
# This also grants DNS Zone Contributor to the add-on's managed identity
az aks approuting zone add \
  --resource-group rg-zava-demo \
  --name aks-zava-demo \
  --ids=${ZONE_ID} \
  --attach-zones
```
> — Source: [Set up a Custom Domain Name and SSL Certificate with the Application Routing Add-on](https://learn.microsoft.com/en-us/azure/aks/app-routing-dns-ssl) | Provenance: adapted

**Step 2: Obtain and import a CA-issued certificate into Key Vault.**

```bash
# Option A: Import an existing CA-issued certificate (PFX format)
az keyvault certificate import \
  --vault-name kv-zava-demo \
  --name tls-cert-ops-agent \
  --file ca-issued-cert.pfx

# Option B: Request a certificate from a Key Vault-integrated CA (e.g., DigiCert)
# This requires prior setup of the CA issuer in Key Vault.
# See: https://learn.microsoft.com/en-us/azure/key-vault/certificates/how-to-integrate-certificate-authority

# Get the certificate URI (needed for the Ingress annotation)
az keyvault certificate show \
  --vault-name kv-zava-demo \
  --name tls-cert-ops-agent \
```

## 5. Workload Identity & Auth from AKS to Foundry

### 5.1 Overview

Microsoft Entra Workload ID is the recommended mechanism for AKS pods to authenticate to Azure services ([Use Microsoft Entra Workload ID with AKS](https://learn.microsoft.com/en-us/azure/aks/workload-identity-overview)):

> "Workloads deployed on an AKS cluster require Microsoft Entra application credentials or managed identities to access Microsoft Entra protected resources, such as Azure Key Vault and Microsoft Graph. Microsoft Entra Workload ID integrates with the capabilities native to Kubernetes to federate with external identity providers."
> — Source: [Use Microsoft Entra Workload ID with AKS](https://learn.microsoft.com/en-us/azure/aks/workload-identity-overview)

> "Workload ID covers the **pod-to-Azure** identity scenario in AKS — how applications running in pods authenticate to Microsoft Entra–protected services."
> — Source: [Use Microsoft Entra Workload ID with AKS](https://learn.microsoft.com/en-us/azure/aks/workload-identity-overview)

The flow is:
1. AKS cluster has OIDC issuer enabled
2. A user-assigned managed identity (UAMI) is created
3. A federated identity credential links the UAMI to a Kubernetes service account
4. The pod uses `DefaultAzureCredential` from `azure-identity` to acquire tokens

### 5.2 RBAC Role for AKS→Foundry Calls

For the Zava demo, the LangGraph ops agent pod on AKS calls the **Foundry Agents V2 / Responses API** on the Foundry resource. The correct RBAC role for this data-plane access is **`Foundry User`** (previously named `Azure AI User`, GUID `53ca6127-db72-4b80-b1b0-d745d6d5456d`).

The companion Foundry V2 research ([research/2026-05-20-foundry-v2.md](./2026-05-20-foundry-v2.md)) verifies this role and explicitly warns against using Cognitive Services roles for Foundry scenarios:

> "Don't assign built-in roles that start with **Cognitive Services**. These roles are designed for accessing AI Services resources directly and don't apply to Foundry scenarios. Similarly, don't use the **Azure AI Developer** role for Foundry work. Despite the name, this role is scoped to Azure Machine Learning workspaces and Foundry hubs."
> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

The Foundry User role provides the data-plane permissions needed to call Foundry Agents and the Responses API:

| Built-in role | Build in project (data actions) | Reader access |
|---|---|---|
| **Foundry User** | ✔ | ✔ |

> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

> **Alternative — direct Azure OpenAI v1 endpoint calls (not Foundry Responses API):** If the ops agent pod calls the Azure OpenAI endpoint directly (e.g., `https://<resource>.openai.azure.com/...`) instead of the Foundry Responses API, use `Cognitive Services OpenAI User` as documented in [Secure access to Azure OpenAI from AKS](https://learn.microsoft.com/en-us/azure/aks/open-ai-secure-access-quickstart). This is **not** the Zava demo's architecture — the Zava demo uses the Foundry Responses API.

## 8. Security Defaults to NOT Skip

Even for a demo, these security practices should be followed ([Best practices for AKS](https://learn.microsoft.com/en-us/azure/aks/best-practices)):

### 8.1 Always Use Trusted HTTPS/TLS for Public Endpoints

**Any public endpoint MUST use HTTPS with a CA-issued (trusted) TLS certificate.** Transmitting shared secrets, HMAC tokens, or API keys over plain HTTP — even for a short-lived demo — is unacceptable. Self-signed certificates are not acceptable for the external customer demo because they require disabling certificate verification on the client, which undermines TLS security guarantees.

Use the **Application Routing + Azure Key Vault + Azure DNS** path described in Section 4.4 to obtain a trusted certificate. The official documentation explicitly warns that self-signed certificates are "considered unsafe for public-facing websites and applications" ([Set up a Custom Domain Name and SSL Certificate with the Application Routing Add-on](https://learn.microsoft.com/en-us/azure/aks/app-routing-dns-ssl)).

### 8.2 Authenticate All A2A Requests

The A2A endpoint on the LangGraph agent **must not be open to the public without authentication**. Options for demo-grade auth (all over HTTPS with trusted TLS):

1. **Shared secret / API key in a header** — the Foundry agent includes a pre-shared key (e.g., `X-API-Key: <secret>`) in every A2A request. The LangGraph agent validates it. Store the key in Azure Key Vault, loaded via Workload Identity.

2. **IP allowlist** — configure the AKS Network Security Group or NGINX annotations to restrict source IPs to the Foundry service's outbound IPs (if known and stable).

3. **HMAC signature** — the Foundry agent signs each request body with a shared secret; the LangGraph agent verifies the signature. More robust than a static API key.

### 8.3 Use Managed Identity (Not Secrets)

> "We recommend using Microsoft Entra Workload ID and managed identities on AKS for Azure OpenAI access because it eliminates the need to manage credentials."
> — Source: [Secure access to Azure OpenAI from AKS](https://learn.microsoft.com/en-us/azure/aks/open-ai-secure-access-quickstart)

- **DO**: Use Workload Identity for all pod-to-Azure communication
- **DON'T**: Store connection strings or API keys in environment variables or ConfigMaps

### 8.4 Disable ACR Admin User

The Bicep template sets `adminUserEnabled: false` on the ACR. AKS authenticates via the `AcrPull` role assignment — no Docker username/password needed.

### 8.5 Use RBAC

AKS RBAC is enabled by default on new clusters. Keep it enabled. Use Kubernetes RBAC to restrict what the service account can do.

### 8.6 Keep Kubernetes Updated

Use a supported Kubernetes version (1.34 or 1.35 as of May 2026). Enable auto-upgrade for the demo period:

```bash
az aks update \
  --resource-group rg-zava-demo \
  --name aks-zava-demo \
  --auto-upgrade-channel patch
```
> — Source: [Automatically upgrade an AKS cluster](https://learn.microsoft.com/en-us/azure/aks/auto-upgrade-cluster) | Provenance: adapted

3. **Not assigning AcrPull** — pods fail to pull images with "unauthorized" errors
4. **Exposing the A2A endpoint without trusted HTTPS** — always use TLS with a CA-issued certificate, even for a demo (Section 4.4)
5. **Using self-signed certificates for the external demo** — requires disabling cert verification, which is a security anti-pattern
6. **Skipping `az aks stop`** — forgotten clusters accumulate VM costs 24/7
7. **Using an EOL Kubernetes version** — always verify with `az aks get-versions` before deploying
8. **Assigning Cognitive Services roles for Foundry access** — use `Foundry User` instead ([RBAC for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))

## 11. Research Limitations

- **Exact VM pricing** for `Standard_D2s_v5` could not be extracted programmatically from the Azure pricing page (JavaScript-rendered). The ~$0.096/hr figure is an approximate estimate from the Azure pricing calculator and should be verified before deployment. The VM specs (2 vCPU, 8 GiB RAM) are officially confirmed by the [Dsv5 size series page](https://learn.microsoft.com/en-us/azure/virtual-machines/sizes/general-purpose/dsv5-series).
- **Kubernetes version currency** — the version table reflects the [AKS Kubernetes release calendar](https://learn.microsoft.com/en-us/azure/aks/supported-kubernetes-versions) as of the report date. AKS versions rotate frequently; always run `az aks get-versions` at deployment time.
- **Application Routing add-on Bicep property** — the `ingressProfile.webAppRouting.enabled` property is documented in the API reference but the official quickstart doesn't show it in Bicep (it uses `--enable-app-routing` CLI flag). The Bicep property was verified from the [2026-02-01 API reference](https://learn.microsoft.com/en-us/azure/templates/microsoft.containerservice/2026-02-01/managedclusters).
- **Container Insights addon name** — documentation refers to both `omsagent` and the newer Azure Monitor agent. The `omsagent` addon profile name is still used in Bicep templates. The newer data collection rule (DCR) approach was not fully explored.
- **Managed Prometheus full Bicep** — deliberately marked as optional and excluded from the default deployment. A full end-to-end Prometheus deployment requires DCR/DCE/Azure Monitor workspace resources, which are available at the [Azure-Samples Bicep template](https://github.com/Azure-Samples/aks-managed-prometheus-and-grafana-bicep). Container Insights is recommended for this demo.
- **Log Analytics ingestion estimate** — the 0.5 GB/day figure is an educated estimate for a small cluster with one workload. Actual ingestion varies based on log volume, verbosity, and Container Insights configuration.
- **Foundry role assignment scope in Bicep** — the Bicep snippet in Section 5.4 uses `scope: foundryAccountResourceId` which is a string parameter. In a real deployment, this may need to be resolved to an `existing` resource reference. The pattern shown follows the standard role assignment approach.
- **DNS zone and certificate provisioning** — the demo requires a real DNS domain. If the user does not own a domain, Azure DNS provides a hosted zone but the domain must be registered separately. Let's Encrypt or DigiCert/GlobalSign via Key Vault integration require domain validation.
