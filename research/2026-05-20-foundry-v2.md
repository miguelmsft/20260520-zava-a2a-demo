## Executive Summary

Microsoft Foundry (branded "Microsoft Foundry," formerly Azure AI Foundry / Azure AI Studio) is the current Azure platform-as-a-service for enterprise AI operations. **Foundry V2** — the "new Foundry" or "new portal experience" — replaced the classic hub-based architecture with a simplified **Foundry resource + project** model. The top-level resource is `Microsoft.CognitiveServices/accounts` with `kind: AIServices`, and projects are child resources (`Microsoft.CognitiveServices/accounts/projects`). This is architecturally distinct from the classic hub + Azure OpenAI model. The new portal (at [ai.azure.com](https://ai.azure.com)) is **generally available** as of early 2026.

For the Zava Smart Order Feasibility demo, Foundry V2 is the correct platform. It supports **Foundry Agents V2** (based on the Responses API, replacing the legacy Assistants API which sunsets August 26, 2026), project-scoped RBAC, and Bicep-based IaC deployment. **Foundry projects are supported in the following target US regions: East US, East US 2, West US, and West US 3 — but NOT West US 2**, which is not listed in the Foundry project region support ([Feature availability across cloud regions](https://learn.microsoft.com/en-us/azure/foundry/reference/region-support)). **The recommended primary region for this demo is East US 2**, as it supports Foundry projects, the Responses API, Agent Service, and all candidate models including `gpt-5.5` via Global Standard deployment.

The RBAC roles were recently renamed from "Azure AI \*" to "Foundry \*" — the key roles are **Foundry Owner** (previously **Azure AI Owner**), **Foundry Account Owner** (previously **Azure AI Account Owner**), **Foundry User** (previously **Azure AI User**), and **Foundry Project Manager** (previously **Azure AI Project Manager**). Role definition GUIDs are unchanged; use GUIDs in automation since some docs/portals may still show previous names during the rollout ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)).

**Model availability for the demo's candidate list:**
- `gpt-5.5`, `gpt-5.4`, and `gpt-5.4-mini` are supported by the Responses API ([Use the Azure OpenAI Responses API](https://learn.microsoft.com/en-us/azure/foundry/openai/how-to/responses)).
- **⚠️ `gpt-5.5-mini` is NOT listed** in either the Responses API model support list or the Foundry model regional availability page — it cannot be used for this demo.
- `gpt-5.4` and `gpt-5.4-mini` are available via Global Standard in all US regions; `gpt-5.5` Global Standard is limited to East US 2 and South Central US ([Foundry model regional deployment availability](https://learn.microsoft.com/en-us/azure/foundry/foundry-models/concepts/models-sold-directly-by-azure-region-availability)).
- **A2A protocol support on Foundry Agents is in Preview** ([Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic)).

The current GA Bicep API version for all Foundry resources (`Microsoft.CognitiveServices/accounts`, `/projects`, `/deployments`) is **`2026-03-01`** per the Azure template reference pages. The official `foundry-samples` GitHub repository still uses `2025-06-01` (a valid prior GA version); this report provides an updated project-ready Bicep variant using `2026-03-01`.

## 1. Overview — What Is Foundry V2 vs. Classic

### What Is Foundry V2 (New Experience) vs. Foundry Classic / Hubs

Microsoft Foundry has undergone several naming and architectural evolutions. The official documentation maps the lineage explicitly:

> | Dimension | Previous | Current |
> |-----------|---------|-----|
> | Brand | Azure AI Studio / Azure AI Foundry | Microsoft Foundry |
> | Portal | Foundry (classic) | Foundry |
> | Agent API | Assistants API (Agents v0.5/v1) | Responses API (Agents v2) |
> | Resource model | Hub + Azure OpenAI + Azure AI Services | Foundry resource (single, with projects) |
> | SDKs & endpoints | Multiple packages against 5+ endpoints | Unified project client (`azure-ai-projects` 2.x) + `OpenAI()` against one project endpoint |
> | Terminology | Threads, Messages, Runs, Assistants | Conversations, Items, Responses, Agent Versions |
>
> — Source: [Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic)

The **architectural difference** is fundamental:

- **Classic (V1 / Hubs):** Used an Azure AI Hub (`Microsoft.MachineLearningServices/workspaces` with kind `Hub`) as a top-level governance container, with Azure OpenAI resources and Azure AI Services as separate resources. Projects were child workspaces of the hub. Multiple resource providers, multiple endpoints.
- **New (V2 / Foundry):** Uses a single `Microsoft.CognitiveServices/accounts` resource with `kind: AIServices` as the top-level Foundry resource. Projects are child resources (`Microsoft.CognitiveServices/accounts/projects`). One resource provider namespace, one project endpoint.

> **Product naming**: Microsoft's AI Platform has evolved from Azure AI Studio → Azure AI Foundry → to Microsoft Foundry (current). Similarly, our AI services portfolio evolved with the platform from Azure Cognitive Services → Azure AI Services → to Foundry Tools (current). Despite the platform evolution, the Azure resource type remains `Microsoft.CognitiveServices/accounts`.
> — Source: [Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic)

### Which Supports Foundry Agents V2

**Only the new Foundry experience** supports Agents V2 (Responses API-based agents). The classic experience uses the legacy Assistants API (Agents v0.5/v1), which sunsets **August 26, 2026**.

> Agents v2 are supported in the new Foundry UI. Existing assistants and v1 agents aren't supported in the new Foundry experience. To use or edit assistants, continue using Foundry (classic) portal until assistant upgrade is available.
> — Source: [New Microsoft Foundry portal general availability overview](https://learn.microsoft.com/en-us/azure/foundry/concepts/general-availability)

### Migration / Coexistence Story

The two experiences coexist. The Foundry portal at [ai.azure.com](https://ai.azure.com) has a **"New Foundry" toggle** in the banner — when on, you see the new project-based experience; when off, you see Foundry (classic). Hub-based projects created in the classic experience remain accessible in the classic portal.

> Using hub-based projects? Hub-based projects are accessible in the [Foundry (classic) portal](../foundry-classic/what-is-foundry.md). New investments are focused on Foundry projects in the new portal.
> — Source: [What is Microsoft Foundry?](https://learn.microsoft.com/en-us/azure/foundry/what-is-foundry)

Key migration dates from official docs:
- **May 30, 2026** — `azure-ai-inference` package retires. Migrate to the `openai` package.
- **August 26, 2026** — Assistants API sunsets. Use Foundry Agents (V2) with the Responses API.

([Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic))

## 2. Key Concepts — Resource Topology

### Resource Topology

Foundry V2 organizes AI workloads through a layered architecture:

> Microsoft Foundry organizes AI workloads through a layered architecture: a top-level Foundry resource for governance, projects for development isolation, and connected Azure services for storage, search, and secrets management.
> — Source: [Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture)

#### Resource Type Table

The exact Azure resource types, as documented:

| Resource type | Resource provider and type | Kind | Supported capabilities |
|---|---|---|---|
| **Microsoft Foundry** | `Microsoft.CognitiveServices/accounts` | `AIServices` | Agents, Evaluations, Azure OpenAI, Speech, Vision, Language, and Content Understanding |
| **Foundry project** | `Microsoft.CognitiveServices/accounts/projects` | `AIServices` | **Subresource to the above** |
| Azure Speech in Foundry Tools | `Microsoft.CognitiveServices/accounts` | `Speech` | Speech |
| Azure Language in Foundry Tools | `Microsoft.CognitiveServices/accounts` | `Language` | Language |
| Azure Vision in Foundry Tools | `Microsoft.CognitiveServices/accounts` | `Vision` | Vision |

> — Source: [Microsoft Foundry architecture — Resource types](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture)

**Important:** The `kind` for a Foundry resource is **`AIServices`**, not `FoundryServices`. This is confirmed in both the architecture docs and the official Bicep templates. The current GA API version for these resources is **`2026-03-01`** per the Azure template reference ([Microsoft.CognitiveServices/accounts template reference](https://learn.microsoft.com/en-us/azure/templates/microsoft.cognitiveservices/accounts)). The official `foundry-samples` GitHub repository uses `2025-06-01` (a valid prior GA version).

#### Resource Hierarchy

> - **Foundry resource**: Top-level Azure resource where you manage governance settings such as networking, security, and model deployments.
> - **Project**: Development boundary inside the Foundry resource where teams build and evaluate use cases. Projects let teams prototype within a preconfigured environment, reusing existing model deployments and connections without repeated IT setup.
> - **Project assets**: Files, agents, evaluations, and related artifacts scoped to a project.
> - **Connected resources**: Azure services such as Storage, Key Vault, and Azure AI Search that the Foundry resource references through connections. These resources have separate governance boundaries, so you manage their networking and access policies independently.
>
> — Source: [Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture)

#### Model Deployments

Model deployments are child resources of the Foundry account (not the project):

- Resource type: `Microsoft.CognitiveServices/accounts/deployments`
- Current GA API version: `2026-03-01` ([Microsoft.CognitiveServices/accounts/deployments template reference](https://learn.microsoft.com/en-us/azure/templates/microsoft.cognitiveservices/accounts/deployments))
- Deployment types: Global Standard, Global Provisioned, Global Batch, Data Zone Standard, Data Zone Provisioned, Data Zone Batch, Standard, Regional Provisioned, Developer

([Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture))

#### Connected Resources

Connected resources (Storage, Key Vault, Azure AI Search) are independent Azure resources with their own governance boundaries. They are referenced via connections, not as child resources.

> Connected resources like Storage, Key Vault, and Azure AI Search are independent Azure resources with their own governance boundaries. You manage networking, access policies, and compliance settings for these resources separately from the Foundry resource.
> — Source: [Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture)

For the basic setup (used in this demo), managed storage is used by default — no customer storage account is required:

> In the default setup, Foundry uses Microsoft-managed storage accounts that are logically separated and support direct file uploads for select use cases, such as OpenAI models and Agents, without requiring a customer-provided storage account.
> — Source: [Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture)

## 3. Getting Started — Bicep Deployment

### Prerequisites

- An Azure subscription
- **Foundry Account Owner** (previously "Azure AI Account Owner") or Azure **Owner** / **Contributor** role to create the Foundry resource
- Azure CLI v2.67.0+ or Bicep CLI installed
- The target region must support the Responses API (see [Section 6 — Regions](#foundry-project-regions-all-supported))

### Installation & Setup

#### Terminal Commands

```bash
# 1. Log in to Azure
az login

# 2. Create a resource group (East US 2 recommended — supports all candidate models)
az group create --name exampleRG --location eastus2

# 3. Deploy Foundry resource + project using Bicep
az deployment group create \
    --resource-group exampleRG \
    --template-file main.bicep \
    --parameters aiFoundryName=myai aiProjectName=myai-proj

# 4. Deploy a model (gpt-5.4-mini Global Standard — verified available in East US 2)
az cognitiveservices account deployment create \
    --name myai \
    --resource-group exampleRG \
    --deployment-name gpt-5.4-mini \
    --model-name gpt-5.4-mini \
    --model-version "2026-03-17" \
    --model-format OpenAI \
    --sku-capacity 10 \
    --sku-name GlobalStandard

# 5. Verify the deployment
az cognitiveservices account deployment show \
    --name myai \
    --resource-group exampleRG \
    --deployment-name gpt-5.4-mini

# 6. List all resources in the group
az resource list --resource-group exampleRG

# 7. Clean up when done
az group delete --name exampleRG
```

### Bicep Template A — Official Sample (Generic, `gpt-4.1-mini`)

The following is adapted from the **official basic Bicep template** in the `microsoft-foundry/foundry-samples` repository. It uses API version `2025-06-01` as published in the sample. This is a valid prior GA version; see Template B below for the current GA `2026-03-01` version.

```bicep
// Source: https://github.com/microsoft-foundry/foundry-samples/blob/main/infrastructure/infrastructure-setup-bicep/00-basic/main.bicep
// Provenance: adapted (comments reformatted, second commented model deployment block omitted)
// Note: Uses API version 2025-06-01 as published in the official sample.

param aiFoundryName string = 'foundry-name'
param aiProjectName string = '${aiFoundryName}-proj'
param location string = 'eastus2'

/*
  A Foundry resource is a variant of a CognitiveServices/account resource type.
*/
resource aiFoundry 'Microsoft.CognitiveServices/accounts@2025-06-01' = {
  name: aiFoundryName
  location: location
  identity: {
    type: 'SystemAssigned'
  }
  sku: {
    name: 'S0'
  }
  kind: 'AIServices'
  properties: {
    allowProjectManagement: true
    customSubDomainName: aiFoundryName
    disableLocalAuth: false
  }
}

/*
  Projects are sub-resources that group work for a use case.
*/
resource aiProject 'Microsoft.CognitiveServices/accounts/projects@2025-06-01' = {
  name: aiProjectName
  parent: aiFoundry
  location: location
  identity: {
    type: 'SystemAssigned'
  }
  properties: {}
}

/*
  Deploy a model for use in playground, agents, and other tools.
*/
resource modelDeployment 'Microsoft.CognitiveServices/accounts/deployments@2025-06-01' = {
  parent: aiFoundry
  name: 'gpt-4.1-mini'
  sku: {
    capacity: 1
    name: 'GlobalStandard'
  }
  properties: {
    model: {
      name: 'gpt-4.1-mini'
      format: 'OpenAI'
      version: '2025-04-14'
    }
  }
}
```

## 4. Core Usage — RBAC

### RBAC — Built-in Roles

Microsoft Foundry uses four **Foundry-specific built-in roles** plus the standard Azure roles (Owner, Contributor, Reader). The Foundry roles were **recently renamed** from "Azure AI \*" to "Foundry \*":

> The Foundry RBAC roles were recently renamed. **Foundry User**, **Foundry Owner**, **Foundry Account Owner**, and **Foundry Project Manager** were previously named Azure AI User, Azure AI Owner, Azure AI Account Owner, and Azure AI Project Manager. You might still see the previous names in some places while the rename rolls out. The role IDs and core permissions are unchanged by the rename.
> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

#### What the Zava Demo User Needs

For the demo user (`miguelmsft`) who needs **full use of the Foundry resource** — creating accounts, projects, deploying models, building agents, and publishing:

1. **Foundry Owner** (previously "Azure AI Owner") at the Foundry resource scope — this is the most privileged self-service role with both control-plane and data-plane access. It is described as:

   > To publish agents, you need the **Foundry Project Manager** role (minimum) on the Foundry resource scope.

#### Minimum Role Assignments to Get Started

> - Assign the **Foundry User** role on your Foundry resource to your **user principal**.
> - Assign the **Foundry User** role on your Foundry resource to your **project's managed identity**.
>
> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

Similarly, don't use the **Azure AI Developer** role for Foundry work. Despite the name, this role is scoped to Azure Machine Learning workspaces and Foundry hubs.
> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

## 5. Configuration & Best Practices

### Recommended Configuration

For the Zava demo (public endpoints, basic setup):

1. **Foundry resource** with `kind: AIServices`, `allowProjectManagement: true`
2. **One project** per agent use case
3. **Model deployments** as Global Standard (for maximum flexibility)
4. **Managed storage** (default — no customer storage needed for basic agent setup)
5. **Foundry Owner** role (previously "Azure AI Owner") for the demo operator
6. **Foundry User** role (previously "Azure AI User") for the project's managed identity

([Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture); [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))

### Best Practices

**Security:**
> RBAC roles apply when you authenticate using Microsoft Entra ID. If you use key-based authentication instead, the key grants full access without role restrictions. Microsoft recommends using Entra ID authentication for improved security and granular access control.
> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

**Role Assignments:**
- Use Microsoft Entra groups for team access rather than individual assignments ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))
- Use role definition GUIDs in code/IaC during the rename rollout ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))
- Scope role assignments at the narrowest level needed (project scope for developers, resource scope for admins) ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))

**Enterprise RBAC Mapping:**

| Persona | Role and Scope | Purpose |
|---|---|---|
| IT admin | Owner on subscription scope | Ensure Foundry resource meets enterprise standards |
| Managers | Foundry Account Owner on Foundry resource scope | Manage resources, deploy models, audit |
| Team lead | Foundry Project Manager on Foundry resource scope | Create projects, assign Foundry User to team |
| Developers | Foundry User on project scope + Reader on resource scope | Build agents with pre-deployed models |

> — Source: [Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry)

### Common Pitfalls & Anti-Patterns

1. **Using `kind: FoundryServices`** — this is NOT the correct kind. Use `AIServices`. ([Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture))
2. **Forgetting `allowProjectManagement: true`** — without this property, project sub-resources cannot be created. ([Quickstart: Set up Microsoft Foundry resources](https://learn.microsoft.com/en-us/azure/foundry/tutorials/quickstart-create-foundry-resources))
3. **Using Cognitive Services roles** — they don't apply to Foundry scenarios. ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))
4. **Not assigning Foundry User to project managed identity** — the project's managed identity needs this role on the Foundry resource. ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))
5. **Deploying in a region that doesn't support the Responses API** — check region availability before provisioning. ([Use the Azure OpenAI Responses API](https://learn.microsoft.com/en-us/azure/foundry/openai/how-to/responses))

## 8. Key Concepts for Decision-Makers

### What Is Foundry V2 and Why Should You Care?

Microsoft Foundry is Azure's unified AI platform-as-a-service that consolidates what was previously scattered across Azure OpenAI, Azure AI Services, Azure AI Studio, and Azure Machine Learning into a single resource model. Instead of managing multiple resource providers, endpoints, and SDKs, you now have one `Microsoft.CognitiveServices/accounts` resource with child projects. This simplification reduces operational overhead, unifies RBAC, and provides a single portal experience. ([Microsoft Foundry architecture](https://learn.microsoft.com/en-us/azure/foundry/concepts/architecture))

For organizations already using Azure OpenAI, upgrading to Foundry keeps existing configurations intact:

> You can upgrade your Azure OpenAI resource to a Foundry resource. You keep your existing Azure OpenAI API endpoint, state of work, and security configurations, and don't need to create a new Foundry resource.
> — Source: [Upgrade from Azure OpenAI to Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/how-to/upgrade-azure-openai)

Specifically, the upgrade preserves:

> Your existing resource configurations and state stay the same, including:
> - Network configurations
> - Access and identity configurations
> - API endpoint and API key
> - Existing state including fine-tuning jobs, batch, stored completions, and more
>
> — Source: [Upgrade from Azure OpenAI to Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/how-to/upgrade-azure-openai)

### The Foundry Resource Model Is Designed for Enterprise Governance

The Foundry architecture separates governance from development. IT administrators manage the top-level Foundry resource (networking, security, model deployments, RBAC), while development teams work within projects that inherit the resource's configuration. This means an admin can set up network isolation, deploy approved models, and configure content safety guardrails once, and multiple project teams benefit without duplicating infrastructure work. RBAC is scoped at both the resource and project level, allowing fine-grained access control — developers get "Foundry User" on their project, team leads get "Foundry Project Manager," and admins get "Foundry Account Owner." ([Role-based access control for Microsoft Foundry](https://learn.microsoft.com/en-us/azure/foundry/concepts/rbac-foundry))

### Agents V2 Is the Future — Assistants API Is Sunsetting

The new Responses API-based Agents (V2) replace the legacy Assistants API, which sunsets August 26, 2026 ([Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic)). Agents V2 are synchronous by default (no polling loops), support versioned agent definitions, and integrate with the new Foundry portal's evaluation, tracing, and monitoring capabilities. For new projects, there is no reason to use the Assistants API — start with Agents V2 directly. Note that **A2A protocol support is currently in Preview** ([Migrate from the Foundry (classic) portal](https://learn.microsoft.com/en-us/azure/foundry/how-to/navigate-from-classic)).

### Bicep Deployment Is First-Class

Microsoft provides official Bicep templates in the `microsoft-foundry/foundry-samples` repository covering multiple scenarios — from basic account+project setup through private networking, customer-managed keys, standard agent setup, and APIM integration ([microsoft-foundry/foundry-samples](https://github.com/microsoft-foundry/foundry-samples)). The `00-basic` template is a three-resource deployment (account, project, model). This IaC-first approach means you can version-control your entire Foundry environment and replicate it across dev/staging/production.

## 9. Research Limitations

1. **`gpt-5.5-mini` not found anywhere.** Neither the Responses API model support list nor the Foundry model regional availability page lists `gpt-5.5-mini`. This model either does not exist, is not yet released, or is named differently. It cannot be used for this demo.

2. **West US 2 not in Foundry project regions.** The region support doc does not list West US 2 as a supported region for Foundry projects, though it lists West US and West US 3. **Do not select West US 2** for the demo.

3. **Role rename rollout in progress.** The documentation explicitly states the rename from "Azure AI \*" to "Foundry \*" is rolling out. CLI commands and portal UI may still show the old names (Azure AI User, Azure AI Owner, Azure AI Account Owner, Azure AI Project Manager) in some places. Role definition GUIDs are unchanged.

4. **Agent Service supported models.** The Agent Service limits page states that Agent Service requires models compatible with the Responses API. Any model listed in the Responses API model support table should work with Agent Service, but per-model Agent Service compatibility is not independently listed — it is inferred from Responses API support. ([Foundry Agent Service limits, quotas, and regional support](https://learn.microsoft.com/en-us/azure/foundry/agents/concepts/limits-quotas-regions))

5. **A2A protocol is Preview.** The feature comparison table on the classic migration page lists A2A protocol status as "Preview." This means A2A-specific behavior, API stability, and regional support may change. This report documents the A2A Preview status but does not independently verify A2A-specific API contracts — that is covered in separate A2A-focused research.

6. **Preview features.** Several portal features are still in Preview (Workflows, Tracing, Monitoring, Knowledge, Memory, Managed VNet, Hosted Agents). The core agent development and model deployment features are GA. ([New Microsoft Foundry portal general availability overview](https://learn.microsoft.com/en-us/azure/foundry/concepts/general-availability))

7. **Official Bicep sample API version.** The `microsoft-foundry/foundry-samples` `00-basic` template uses API version `2025-06-01`. The current GA version per Azure template reference is `2026-03-01`. The sample has not yet been updated. This report provides a project-ready Bicep variant using `2026-03-01`.

8. **Standard/Regional deployment type.** The Standard/Regional availability table on the model availability page does not list `gpt-5.4`, `gpt-5.4-mini`, or `gpt-5.5` for US regions — only Global Standard and Data Zone Standard are available for these models in the US. ([Foundry model regional deployment availability](https://learn.microsoft.com/en-us/azure/foundry/foundry-models/concepts/models-sold-directly-by-azure-region-availability))
